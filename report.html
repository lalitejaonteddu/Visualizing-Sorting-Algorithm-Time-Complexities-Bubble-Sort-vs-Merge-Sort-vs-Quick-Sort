Executive Summary

This project implements and benchmarks five fundamental sorting algorithms in Python, providing an interactive Streamlit web application that visualises both theoretical complexity classes and real measured performance on user-supplied inputs.

Each algorithm is instrumented to count comparisons, array writes / moves, and wall-clock time in milliseconds. Results are compared against theoretically derived operation estimates to validate asymptotic predictions in practice.

Key insight: For large n, quadratic algorithms (Bubble, Insertion, Selection Sort) degrade significantly. At n = 1000, n² = 1,000,000 operations vs n log₂ n ≈ 9,966 for the divide-and-conquer algorithms — a ~100× difference.

Implementation Highlights

SortResult Dataclass — algorithms.py

All five algorithms return a unified, immutable result object:

@dataclass(frozen=True) class SortResult: name: str sorted_values: list[int] comparisons: int swaps: int # "swaps" = array writes / moves elapsed_ms: float

Instrumented Quick Sort — algorithms.py

def quick_sort(values: list[int]) -> SortResult: arr = values.copy() comparisons = swaps = 0 start = perf_counter() def partition(a, low, high): nonlocal comparisons, swaps pivot = a[high] # last-element pivot i = low - 1 for j in range(low, high): comparisons += 1 if a[j] <= pivot: i += 1; a[i], a[j] = a[j], a[i]; swaps += 1 a[i+1], a[high] = a[high], a[i+1]; swaps += 1 return i + 1

Theoretical Estimator — complexity.py

def theoretical_operation_estimates(n, already_sorted): bubble = n - 1 if already_sorted else n*(n-1)//2 merge = int(n * math.log2(n)) insertion = n - 1 if already_sorted else n*(n-1)//2 selection = n*(n-1)//2 # always quadratic quick = int(n * math.log2(n)) return {"bubble_comparisons": bubble, ...}
Design decision: Separating algorithms.py (pure sorting logic) from complexity.py (theory + parsing) and app.py (UI) makes the codebase testable and reusable independently of Streamlit.

Key Findings & Conclusions

1. Empirical results align with theory. Quick Sort and Merge Sort consistently require far fewer comparisons than the O(n²) algorithms, even at small n. At n = 20, they used ~32× fewer comparisons than Selection Sort.
2. Constant factors matter at small n. Merge Sort had more overhead than Quick Sort (0.0175 ms vs 0.0084 ms) despite similar comparison counts, due to list allocation in its recursive merge step.
3. Selection Sort's write efficiency. With only 17 array writes (vs 113 for Bubble/Insertion), Selection Sort is the best choice when write operations are costly — such as writing to flash memory or mechanical drives.
4. Input order matters. The app detects pre-sorted input and adjusts theoretical estimates accordingly — Bubble Sort and Insertion Sort degrade to O(n) on already-sorted data, while Selection Sort remains O(n²) regardless.

When to Use Each Algorithm

Algorithm Use When… Avoid When…
Bubble Sort Teaching / demonstration only n > 1000; any production use
Insertion Sort Nearly sorted data; n < 50 Large random datasets
Selection Sort Minimising write operations Large n; stability required
Merge Sort Stable sort required; large n Memory-constrained environments
Quick Sort General purpose; cache-efficient Adversarial / already-sorted input (unmitigated pivot)
